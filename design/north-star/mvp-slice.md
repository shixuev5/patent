# MVP 切片设计（议题 F）

**目的**：把北极星切成可在 6 个月内交付、可被真实审查员使用、可被客观度量的最小版本。
**关系**：本文件假设议题 G（客户访谈）已经完成或同步进行；G 的结论会反向修正本文。

---

## 1. MVP 要证明的"一句话命题"

> **在 [AI/软件领域 · 新申请的第一次 OA 阶段]，AI 可以自主完成 95% 以上的检索 + claim chart + 创造性判断 + OA 起草，审查员只在分歧/低置信/高影响处裁决；总体审查质量不下降，单件平均时长下降 ≥ 50%。**

这是**一个可证伪的命题**。如果做完 MVP 命题不成立，北极星方向需要重新考虑——这就是 MVP 的价值。

---

## 2. 从北极星砍掉什么、保留什么

### 砍掉（v2 以后再做）

| 功能 | 砍的理由 |
|------|---------|
| **Artifact Canvas 深查模式** | 编辑 OA 文书是重投入功能；MVP 用「外部 Word 导出 + 复制粘贴」临时方案 |
| **Policy Console 完整版** | 政策学习需要积累数据；MVP 用「预置的处室级标准 prompt」替代审查员自维护 |
| **Ambient Insight** | Nice-to-have；MVP 不影响核心闭环验证 |
| **时限看板** | 现有审查系统已有，不重做 |
| **对话历史跨案件视图** | 单案件内的对话保留；跨案件检索 v2 |
| **AI-to-AI Negotiation** | 长期愿景，与申请人侧 agent 协议依赖外部生态成熟 |
| **多 agent 投票分歧机制** | MVP 用单 agent + confidence 阈值；多 agent 是 v2 升级 |
| **复审 / 异议流程** | 不在 MVP 范围 |

### 保留（核心闭环）

| 功能 | 必须保留的理由 |
|------|--------------|
| **Decision Queue** | 这是产品形态的核心差异化；砍了就退化成 Copilot |
| **Reasoning Card** | 证据链可审计是 H4 的核心检验；砍了就没法签字 |
| **置信度透明 + 一键裁决** | 北极星交互的最小单元 |
| **审计 / 抽样复核** | 法律合规要求；可降级但不可缺 |
| **AI 自主跑完检索+判断+起草** | 没这个就没有 MVP 价值；这是 95% 工作量 |

## 3. 范围边界

### 3.1 领域：IPC G06F + G06N（AI / 软件）

选这个领域的原因：
- 该领域 AI 类申请占比高（2024–2025 中国 AI 专利申请 +127%），样本量大
- 审查员相对接受 AI 工具
- 现有技术多为数字文献，AI 检索覆盖率高
- 我们自己有更深的领域理解

### 3.2 案件阶段：新申请 → 第一次 OA

完整的审查生命周期可分多个阶段，MVP 只覆盖最简单一段：

```
[申请受理] → [初审] → ★ [实审 · 第一次 OA] ★ → [申请人答复] → [二次 OA / 授权 / 驳回] → [复审]
                                ↑
                             MVP 在这里
```

理由：
- 这是最高频、最标准化的环节
- 输入清晰（一份说明书）、输出清晰（一份 OA 或授权建议）
- 失败成本低（OA 可由人改了再发，不是不可逆决定）
- 涉及法条少（主要 A22.3 创造性、A26 公开充分性）

### 3.3 法条范围：A22.3 + A26.3 + A26.4

只覆盖最常用三条。其他法条（如 A2.2 专利保护客体、A33 修改超范围）在 MVP 中 **fallback 到人工审查**。

### 3.4 用户范围：5–10 名试点审查员

- 优先 CNIPA 北京 / 苏州 审协中心 AI 领域审查员
- 退而求其次：与某省 IP 服务机构合作的"实质审查辅助"项目
- 最后兜底：找 3–5 名前审查员（已离职到代理所）做模拟测试

### 3.5 部署模式：Shadow Mode（v1）→ Co-pilot Mode（v1.5）

**Shadow mode（前 3 个月）**：
- AI 在后台跑完整流程，结果**不进入正式审查档案**
- 审查员正常按既有流程工作
- 每天结束后，审查员可以对比：AI 的结论 vs 自己的结论
- 我们采集**判决一致性数据**作为核心指标
- 风险接近零，纯学习模式

**Co-pilot mode（第 4–6 个月）**：
- AI 结论可被审查员**采纳为正式 OA 草稿**（仍需审查员签字）
- 上线 Decision Queue 作为入口
- 仍是"AI 提议 + 人决定"，不是 Disagreement-first 的完整态

> ⚠ **MVP 不上线"AI 自主执行"模式**。Disagreement-first 的完整体验需要在客户深度信任后才能尝试，且需要 CNIPA 政策松动。这一步至少推迟到 v2（12 个月后）。

## 4. 技术架构最小版

```
┌─────────────────────────────────────────────────────┐
│            Web Frontend (React + Tailwind)          │
│      Decision Queue · Reasoning Card · Sign-off     │
└──────────────────────┬──────────────────────────────┘
                       │ API
┌──────────────────────▼──────────────────────────────┐
│             Backend (Python + FastAPI)              │
│   Orchestrator · Case State · Audit Log · Auth     │
└─────┬──────────┬──────────┬──────────┬───────────────┘
      │          │          │          │
      ▼          ▼          ▼          ▼
   检索        Claim       创造性       OA
  Agent       Chart       判断         起草
  (semantic  Agent        Agent        Agent
   search)
      │          │          │          │
      └──────────┴──────────┴──────────┘
                       │
                       ▼
              Claude Sonnet 4.6
              (主要推理)
              + Opus 4.7 (创造性判断)
```

**复用的现有资产**：本 repo 的 `backend/ai_search/` 已经有相关检索能力，MVP 第一版可以直接拿来用。

**新增最小工程量**：
- Orchestrator 编排逻辑
- Case state machine（4–5 个状态）
- Audit log 持久化（合规要求）
- 一致性度量管线（shadow mode 用）
- Decision Queue UI + Reasoning Card UI

## 5. 时间线 + 里程碑

> 总时长 6 个月。前 1 个月与议题 G（客户访谈）并行。

| 月 | 里程碑 | 验证 |
|----|--------|------|
| **M1** | 完成客户访谈 #1–#4 · 招募 2 名试点审查员 · 后端 prototype（单 agent 跑通一个案件）| 假设 H1–H3 |
| **M2** | 4 个 agent 并联可工作 · 前端 Decision Queue + Reasoning Card 接通真实数据 · 开始 shadow mode | 技术可行性 |
| **M3** | Shadow mode 跑满 1 个月 · 收集 ≥ 50 个案件的"AI vs 审查员"一致性数据 | 命题中段验证 |
| **M4** | 进入 Co-pilot mode · 至少 2 名审查员采纳 ≥ 5 件 AI 起草 OA | H4 验证 |
| **M5** | 扩到 5–10 名审查员 · 度量平均审查时长 + 质量（被复审推翻率）| 核心命题验证 |
| **M6** | 总结 · 命题成立则规划 v2；不成立则改弦更张 | Go / no-go |

## 6. 成功指标 / Kill criteria

**Primary metric（核心命题验证）**：

| 指标 | 成功标准（M5–M6 测） | 不达则 |
|------|-------------------|--------|
| AI 与审查员一致率（shadow mode） | ≥ 80% | 调 agent prompt / 重训练，3 个月给一次机会 |
| 试点审查员单件平均时长 | ≤ 历史均值 50% | 没价值，命题不成立 |
| 试点审查员对 MVP 的 NPS | ≥ 7（满分 10）| 产品方向有问题 |
| 试点期间被复审推翻案件 | ≤ 历史推翻率 | **质量下降是致命**，必须 ≤ 原水平 |

**Secondary metrics**：
- Reasoning Card 内一键采纳比例（衡量 AI 直接可用程度）
- 审查员追问 agent 的次数（衡量是否还需要做"对话深查"）
- 审查员对 Confidence 数值的信任程度（问卷）

**Kill criteria（任一触发，停止 MVP 投入重新评估）**：
- 🔴 Shadow mode 3 个月，一致率 < 60%
- 🔴 试点审查员中 ≥ 50% 在 M3 后退出
- 🔴 任一被采纳的 AI OA 在 M4–M6 被复审推翻
- 🔴 CNIPA / 主管机构在 M2–M4 明确禁止该试点

## 8. 与北极星形态的差异（视觉对比）

| 维度 | 北极星 | MVP |
|------|--------|-----|
| **核心交互** | Disagreement-first（AI 主动只在分歧找人）| Co-pilot（AI 全程提议，人全程决定） |
| **AI 自主度** | 95%+ 案件 AI 自主完成 | 0%（shadow）→ 100% 案件 AI 提议但人审批 |
| **签字** | 高置信案件一键签 | 每件都需要人审 + 签 |
| **视图数量** | 7 个视图 + 2 个模态 | 2 个视图（Queue + Card）|
| **Policy 维护** | 审查员显式 + agent 提炼双轨 | 预置 prompt，不开放维护 |
| **多 agent 投票** | 是 | 否（单 agent + confidence）|
| **跨案件能力** | ⌘K · Ambient · 对话历史 | 无 |
| **部署边界** | 全流程、全法条、全领域 | 1 个领域、3 个法条、1 个阶段 |

**MVP 是"通向北极星的第一块踏脚石"**，不是北极星的小型版本。它在形态上更保守（Co-pilot），但**用真实环境验证最关键的几个假设**，从而决定下一步是否值得投入。
